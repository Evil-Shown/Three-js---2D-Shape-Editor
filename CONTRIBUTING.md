# Contributing Guide

Thank you for your interest in improving this project.

This repository is an independent personal R&D project. Contributions are welcome when they align with the project's goals, architecture, and professional standards.

## Contribution Principles

- Keep the system generic, reusable, and standalone.
- Preserve modular design between frontend editor logic and optional backend adapters.
- Prefer maintainable, well-tested, and clearly documented changes.
- Avoid introducing business-specific assumptions into architecture or naming.

## How To Contribute

1. Fork the repository and create a feature branch.
2. Keep pull requests focused on a single concern.
3. Include clear commit messages and concise implementation notes.
4. Update documentation when behavior or interfaces change.
5. Run local checks before opening a pull request.

## Coding and Documentation Expectations

- Follow existing project structure and style.
- Do not include unnecessary dependencies.
- Ensure any new API or workflow remains generic and environment-agnostic.
- Add or update tests when introducing behavior changes.

## Compliance and Confidentiality Requirements

All contributors must confirm that submitted changes:

- Do not contain proprietary, confidential, or employer-owned code.
- Do not include internal company references, private endpoints, or restricted assets.
- Are contributed with full legal rights to share under this repository's license.

Any pull request that violates these requirements may be declined.

## Pull Request Checklist

- [ ] Scope is clear and limited
- [ ] Documentation updated if required
- [ ] Local tests/checks completed
- [ ] No proprietary or confidential content included
- [ ] Change remains generic and reusable

## Review Process

Maintainer review evaluates:

- Technical correctness
- Architectural fit
- Documentation quality
- Compliance with confidentiality and licensing requirements

The maintainer may request revisions before merge.
